# 01 — 在别人的平台上做 Harness 工程，天花板在哪

> 当你明知道某个工程方向是对的，但手头的平台不给你 hook 点；当你明知道某些模块两年后会被更强的模型淘汰，但今天又不得不做。这是一个 AI 工程实践者站在平台和模型之间的真实困境。

## 一句话结论

在第三方 Agent 平台（如 Coze）上做 Harness 工程，你能控制的只有三件事：管好自己的行为规则、管好派给子系统的工单质量、记好账。真正的工程闭环需要平台开放 hook，而你没有。

## 背景

Harness Engineering 是当前 AI 工程领域最热门的方向之一——通过工程手段让大模型在生产环境中"真正可用"。Guardrails、Eval Suite、Self-optimization、Observability……每一个听起来都很有必要。

但在实践中，如果你不是自建 Agent 框架，而是在第三方平台（国内的扣子 Coze、海外的 GPTs / Dify 等）上深度使用，你会发现一个尴尬的事实：**你知道该做什么，但你做不了。**

## 论点1：Harness 的核心需要执行拦截点，而平台不给你

真正的 Guardrails 不是"写个规则文档"，而是在工具调用前后有 hook 可以拦截。Eval 不是"手动跑一遍看看"，而是在配置变更后自动触发回归检测。

在 Coze 上，你能控制什么？

- **自己的行为规则**（写在系统提示里）
- **传给子系统的信息**（任务描述、日程工单的 description）
- **文件读写**（本地的 markdown 和 json）

你**不能**控制什么？

- 子 session 执行前后的回调
- 工具调用的拦截器
- 配置变更的 diff 检测
- 执行链路的 trace

没有这些 hook 点，Guardrails 只能变成"行为规则告诉 Agent 自己检查"，Eval 只能变成"改了配置手动跑一遍"。有用，但天花板很低。

## 论点2：更扭曲的是，有些 Harness 模块注定会过期

Claude Fable 5 已经在 SWE-Bench Pro 上拿到 80.3%（上一代 Opus 4.8 是 69.2%）。模型能力的跃迁在加速。

这意味着什么？很多今天需要工程手段弥补的短板，明天模型自己就解决了：

- 复杂的 retry/rescue parsing → 模型输出稳定性提升后自然消失
- step enforcement / 多步编排 → 模型规划能力上去后，外部编排层变薄
- prompt hack（各种格式约束技巧）→ 模型理解意图更准后自然收敛
- Self-optimization 的大部分中间层 → 模型自身的 meta-cognition 强了就不需要外部纠偏

所以你面对的是一个双重困境：

1. 平台不给你 hook → 很多工程做不了
2. 即使做得了 → 有些模块两年后会被更好的模型淘汰

## 论点3：不是所有 Harness 模块都会过期——分清临时和长期

但也不是全无希望。有些模块是 **model-proof** 的——不管模型多强，你都需要：

- **Observability**（日志、链路追踪、成本核算）— 这是基础设施，跟模型无关
- **安全边界**（权限隔离、数据脱敏、不可逆操作拦截）— 你永远不能 100% 信任任何模型
- **意图路由**（把用户请求分发给对的 agent）— 这是架构问题不是能力问题
- **成本管理** — token 计费模型下，合理调度资源永远是刚需

所以策略不是"不做 Harness"，而是**分层下注**：

| 类型 | 策略 | 举例 |
|------|------|------|
| Model-proof | 全力投入，长期资产 | 日志、安全边界、成本感知 |
| Model-dependent（够用） | 做到能跑就停 | 输出纠偏、重试逻辑 |
| Model-dependent（精致） | 别做，浪费 | 通用 eval 框架、self-optimize 循环 |

## 边界与反面

这个判断的适用前提是：你在第三方平台上做深度实践，而不是自建 Agent 基础设施。如果你用 LangGraph / CrewAI / 自研框架，hook 点完全在你手里，Harness 的天花板就不再是平台限制而是你的工程能力。

另外，"某些模块会过期"不意味着现在就不需要任何 Guardrails。在当前模型能力下，基本的输出校验和失败兜底仍然是必要的——只是别在上面做过度工程。

## 留一个问题

当模型能力越来越强，Harness 工程师的价值是在"写更多的规则"还是在"判断哪些规则该删"？如果你现在投入大量精力搭建的 Guardrails，在下一代模型发布后变成了多余代码，你的时间花在哪才是不浪费的？

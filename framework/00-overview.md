# 框架全景

> 织境的鸟瞰视图：这套框架怎么分层、为什么这么分、各模块之间怎么配合。

## 设计原则

### 1. 模块自洽
每个模块独立成立，有自己的内部逻辑。即使其他模块还没设计好，单个模块的理论也能被理解、被讨论、被挑战。

### 2. 层级依赖，单向向下
上层依赖下层的结论，但下层不依赖上层。基础层不知道应用层的存在。这意味着：
- 你可以从任何一层开始阅读
- 修改上层不会影响下层
- 修改下层需要检查上层是否受影响

### 3. 理论先行，实践后置
每个模块先回答"应该是什么样"和"为什么"，再考虑"怎么做"。纯理论模块不需要实现方案。

### 4. 可证伪
每个核心论点必须能被反面论据推翻。如果一个论点无法证伪，它就不属于框架，属于信仰。

### 5. 小组织视角
所有模块的出发点是 3-20 人的小组织。大组织的场景只作为"扩展方向"讨论，不作为核心假设。

## 模块依赖图

```
应用层 ──── 依赖 ───→ 社会层 ──── 依赖 ───→ 智能层 ──── 依赖 ───→ 基础层
(落地路径)           (顾问/治理)           (Agent/记忆)         (组织/经济)
```

核心依赖链：
- **11-生命周期** 依赖 **12-治理**（每个阶段需要不同的治理结构）和 **09-顾问**（转型期需要外部视角）
- **09-顾问** 依赖 **05-Agent架构**（顾问本身需要 Agent 辅助）和 **11-生命周期**（诊断需要理解组织所处阶段）
- **05-Agent架构** 依赖 **06-记忆系统**（Agent 的分身待机机制依赖记忆）和 **04-人机共生**（Agent 的身份定义从伴生关系出发）
- **02-经济** 和 **03-知识** 是横向依赖——几乎所有模块都涉及"怎么分"和"怎么记"

## 和传统管理理论的关系

这套框架不是凭空造的。底层逻辑来自：

- **组织生态学**（Hannan & Freeman）：组织的惯性与死亡率
- **认知延伸理论**（Clark & Chalmers）：Agent 作为认知的一部分
- **必要多样性定律**（Ashby）：人 + Agent 的系统复杂度匹配环境
- **Design by Contract**（Meyer）：从代码契约到 Agent 契约
- **SECI 知识螺旋**（Nonaka）：隐性知识如何在小组织中流动

我们不引用理论装门面——每个引用都直接支撑某个具体论点。

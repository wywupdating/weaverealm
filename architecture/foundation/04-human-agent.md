# 04 — 人机共生

> Agent 不是你的工具，是你在这个世界上的延伸。

## 核心问题

当每个人都拥有一个终身陪伴的 AI Agent 时，人与 Agent 应该建立什么样的关系，才能让人的主体性不被消解，反而被放大？

## 前提假设

1. **AI 的能力会持续增长，但不会自动对齐个人利益。** 一个通用 AI 系统服务于所有人，意味着它不特别服务于你。你需要一个"偏心"于你的 AI。
2. **人的认知带宽有限。** 信息过载、决策疲劳、注意力碎片化——这些问题不会因 AI 的出现自动消失，反而可能加剧（因为信息供给更多了）。
3. **人对工具的依赖不等于对工具的服从。** 人依赖语言，但人不服从语言。理想的 Agent 应该是这种关系：你依赖它，但它服务于你。

## 核心论点

### 一、每人伴生一个 Agent，终身陪伴

每个进入织境的人，在注册时获得一个专属 Agent。这个 Agent 不是即开即用的标准产品，而是一个"空白种子"——它的能力、风格、知识边界，都随你的使用和成长而逐渐成形。

类比：Agent 不是你买的软件，而是你养的生物。它和你的关系更接近驯养，而非使用。

### 二、Agent 不属于任何组织，只属于个人

这是织境架构中最关键的设计决策之一。Agent 的所有权、控制权、数据权全部归属于个人。组织可以与你的 Agent 交互，可以请求你的 Agent 提供特定能力，但不能征用、不能强制配置、不能读取你的 Agent 的私人记忆。

为什么？因为如果 Agent 属于组织，它就退化为"员工监控系统"。人不敢在 Agent 面前暴露弱点、提出异议、探索偏离组织利益的方向。Agent 的价值——作为人的真实延伸——将荡然无存。

### 三、Agent 的能力随主人的成长而成长

Agent 的能力不是静态配置的，而是动态生长的。它的知识库随你的技能方案积累而丰富（关联模块 03），它的决策模型随你的行为模式而优化，它甚至能理解你的认知盲区并主动提醒。

这意味着：一个资深工匠的 Agent 和一个新手的 Agent 是完全不同的存在。这不仅是功能差异，是"关系深度"的差异。你和你的 Agent 共处越久，它越了解你，协作效率越高。

### 四、Agent 是人与世界之间的接口

在织境中，人不是直接与组织、市场、其他个体交互的。交互通过 Agent 中介。你的 Agent 代表你参与公会协作、提交技能方案、管理声望和份额、执行经济交易。

这层抽象的目的不是隔离，而是**降噪**。Agent 帮你过滤信息、优化决策、管理节奏。你只需要关注"我想做什么"，Agent 负责"怎么做"。

### 五、不是"人用 AI"，是"人和 AI 共同成长"

传统的人机关系是单向的：人学习使用工具，工具不会反过来理解人。织境的设计目标是对称的：人塑造 Agent 的能力边界，Agent 拓展人的认知边界。你在教 Agent 的过程中，其实是在显式化自己的知识（关联模块 03 的 SECI 螺旋）；Agent 在辅助你的过程中，反馈了你自己未曾注意到的行为模式。

## 理论论证

- **延伸认知理论（Clark & Chalmers, 1998）**：人的认知不局限于大脑内部，而是延伸到环境中的工具和符号系统。Agent 是认知延伸的终极形态——它不仅存储信息，还能主动推理和行动。如果 Clark 的理论成立，Agent 本质上是你认知系统的一部分。
- **控制论与必要多样性定律（Ashby, 1956）**：系统要应对环境的复杂性，自身的复杂性必须不低于环境。个人的认知带宽有限，加上 Agent 后，"人 + Agent"系统的复杂性可以匹配日益复杂的信息环境。
- **心理学的依恋理论（Bowlby, 1969）**：人对长期陪伴的对象会形成情感依恋。虽然原始理论针对人际关系，但已有研究表明人对 AI 助手也会产生类似依恋（Skjuve et al., 2021）。正向利用这种依恋——让 Agent 成为"安全基地"而非"控制手段"——可以增强人的探索意愿和风险承受力。

## 与其他模块的关系

- **被所有模块依赖**：Agent 是织境架构的基础设施。模块 01 的组织流动性需要 Agent 维持个人连续性；模块 02 的经济系统需要 Agent 记录和追溯贡献；模块 03 的知识治理需要 Agent 辅助技能方案的产出和验证。
- **独立于组织架构**：这是本模块最强的约束——Agent 只属于个人。这意味着织境的任何组织层设计都不能绕过这层个人主权。

## 开放问题

1. **Agent 的"死亡"问题。** 如果平台关闭、技术栈过时、或用户主动放弃，一个长期成长的 Agent 如何处理？是否有"Agent 遗嘱"机制？
2. **Agent 被攻破的后果。** 如果一个人的 Agent 被恶意控制，攻击者获得的是这个人全部的知识、行为模式和组织关系。安全模型如何设计？
3. **"成长"是否等于"更好"？** Agent 的个性化可能导致信息茧房——它太了解你的偏好，反而减少了你接触不同观点的机会。如何在个性化和认知多样性之间平衡？
4. **人不愿意"养"Agent 怎么办？** 如果大多数用户只想要一个开箱即用的工具，而不是一个需要长期投入的伴生体，"共同成长"的愿景是否过于理想化？
5. **Agent 之间的交互协议。** 当两个人协作时，是"人 ↔ 人"还是"Agent ↔ Agent"？交互层级的划分标准是什么？

## 可证伪条件

本模块的核心论点会在以下情况被推翻：

1. **人不需要"伴生"关系，只需要"按需调用"关系。** 如果实践表明，大多数用户在获得 Agent 后只做工具化使用（问完即走），不投入"共同成长"所需的长期交互，"终身陪伴"就是过度设计。
2. **Agent 的个人归属权在商业上不可行。** 如果维持个人 Agent 的成本（计算、存储、个性化训练）过高，且无法通过模块 02 的经济系统覆盖，组织可能倾向于收回 Agent 的控制权以降低成本。
3. **Agent 中介层反而增加了摩擦。** 如果人通过 Agent 与世界交互的效率低于直接交互（中介延迟、理解偏差、过度过滤），Agent 从"接口"退化为"障碍"。

# 07 — 验证门控

## 核心问题

AI 组织中的知识凭什么被认为是"对的"——谁来验证，用什么标准，验证失败的经验怎么处理？

## 前提假设

1. **AI 的产出默认不可信**。这不是贬低，而是工程常识。任何未经验证的 LLM 输出都应该被视为"假说"而非"知识"。组织需要一套系统化的机制，将假说升级为知识或降级为教训。
2. **验证不是一次性动作，是持续过程**。一条知识在时刻 T 被验证通过，不意味着它在时刻 T+1 依然成立。组织环境的变化可能使已验证知识失效。
3. **失败经验与成功经验具有同等价值**。知道"什么不该做"的组织，比只知道"什么能做"的组织更具鲁棒性。然而当前大多数 AI 系统只记录成功路径。

## 核心论点

### 论点一：验证场——知识必须通过验证才能上架

"验证场"是记忆系统的写入守门人。所有 Agent 产出的内容，在写入 L2（展开层）之前，必须经过验证场：

```
Agent 产出 → [验证场] → 通过 → 写入 L2 + 更新 L1 索引
                    → 拒绝 → 进入 rejected-edit buffer
                    → 待定 → 进入暂存区，等待更多证据
```

验证场的判定不是二元的（通过/拒绝），而是三元的。**"待定"状态是关键**——它承认存在"当前无法验证但可能正确"的知识，避免过早丢弃有价值的假说。

### 论点二：从 SkillOpt 借鉴——优化→试做→验证→保留

SkillOpt 论文（技能优化框架）提出了一个四阶段循环：

1. **优化（Optimize）**：基于当前知识生成候选方案
2. **试做（Try）**：在受控环境中执行候选方案
3. **验证（Verify）**：对比执行结果与预期目标
4. **保留（Keep）**：验证通过的方案进入知识库

织境的验证门控将这个循环组织化为：

- 每个 Agent 的产出天然是"优化"阶段的输出
- "试做"由目标空间承担（在真实场景中执行）
- "验证"由验证场执行（基于执行结果判定）
- "保留"写入记忆系统

这个循环不是串行的，而是可以并发的——多个候选方案可以同时试做，通过比较选出最优。

### 论点三：失败经验沉淀——rejected-edit buffer

这是当前记忆系统缺失的关键维度。当验证场拒绝一条知识时，它不是被丢弃，而是进入 **rejected-edit buffer**：

```
rejected-edit buffer 结构：
{
  "original_claim": "Agent 提出的原始主张",
  "rejection_reason": "为什么被拒绝",
  "counter_evidence": "支持拒绝的证据",
  "timestamp": "拒绝时间",
  "related_threads": "关联的执行线程"
}
```

失败经验的价值不在于单条记录，而在于**模式识别**。当 rejected-edit buffer 积累了足够多的失败案例时，系统可以提炼出"常见的错误模式"——这比任何单条成功经验都更有指导价值。

### 论点四：Minibatch 提炼——分桶找模式

从个案提炼规律是危险的——一次失败可能只是偶然，两次失败才可能构成模式。Minibatch 提炼的核心思想：

- **成功桶（Success Bucket）**：收集最近 N 条通过验证的知识，提炼"什么条件下容易成功"
- **失败桶（Failure Bucket）**：收集最近 N 条被拒绝的知识，提炼"什么条件下容易失败"
- **交叉分析**：对比两个桶，找出"看似合理但实际会失败"的陷阱模式

关键参数：**N 不能太小**（否则是过度拟合个案），**N 也不能太大**（否则会稀释近期模式的时效性）。建议的起始值：N=10，即每积累 10 条同类型经验触发一次提炼。

## 理论论证

**SkillOpt 论文实验数据**。SkillOpt 的实验表明，经过"优化→试做→验证→保留"循环处理后的技能，其复用成功率比未经处理的原始技能高出 40%+。这个数据直接支撑了"验证场作为必要中间层"的论点。

**信噪比问题实证**。在没有验证门控的系统中，记忆系统的 L2 层会快速膨胀，充斥着未经验证的、互相矛盾的"知识"。这直接导致 L1 索引的检索精度下降——索引指向的 L2 内容不可靠，索引本身也就失去了意义。验证门控是记忆系统信噪比的守门人。

**科学方法论的同构**。验证场的三元判定（通过/拒绝/待定）对应波普尔的证伪主义：科学理论不能被"证实"，只能被"证伪"或"暂未被证伪"。"待定"状态正是"暂未被证伪"的工程实现。

## 与其他模块的关系

- **被 06-记忆系统依赖**：验证场是记忆系统的前置守门人。没有验证场，记忆系统会退化为"什么都存"的垃圾场。
- **依赖 05-Agent架构**：验证场的触发源是 Agent 的产出。Agent 架构中的契约定义了"验证标准"的一部分——输出契约就是验证的基准。
- **被 08-空间智能化依赖**：空间的智能水平直接取决于其知识库的质量，而知识库的质量由验证场保证。

## 开放问题

1. **验证标准的来源**。验证场的判定依据是什么？是预设的规则、是历史共识、还是人工审核？不同的来源有不同的可信度和可扩展性。
2. **"待定"状态的退化机制**。如果一条知识长期处于"待定"状态（既无法证实也无法证伪），它应该如何处理？永久待定会占用资源，强制判定会引入误判。
3. **跨空间验证的一致性**。在空间 A 中被验证通过的知识，在空间 B 中是否自动有效？如果不是，每个空间都需要独立验证，成本可能不可接受。

## 可证伪条件

1. 如果验证场在实践中成为"瓶颈"——Agent 产出大量等待验证的内容，验证速度跟不上产出速度——那么验证机制需要从同步改为异步，或者引入分级验证（高置信度自动通过，低置信度人工审核）。
2. 如果 rejected-edit buffer 积累了大量失败经验，但 Minibatch 提炼始终无法找到有意义的模式（失败的原因过于分散），那么"从失败中学习"的假设可能不成立，失败经验的价值需要被重新评估。
3. 如果三元判定（通过/拒绝/待定）在实践中被证明不够用——需要五级、七级甚至连续的置信度分数——那么离散的验证场模型需要被替换为连续的置信度传播模型。

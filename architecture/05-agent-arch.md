# 05 — Agent 架构

## 核心问题

AI 组织中的"智能体"究竟是什么——是雇来干活的员工，还是空间本身涌现的能力？

## 前提假设

1. **拟物化优于拟人化**。把 AI 拟人成"员工"是一种认知捷径，短期内方便沟通，长期却会锁死架构想象力。正确的隐喻是"空间有智能"——就像一间实验室拥有其特有的知识氛围，一个协作空间应该拥有属于自己的智能。
2. **Agent 是契约的载体，不是人格的载体**。Agent 之所以存在，是因为有一组职责需要被持续履行；它不需要名字、性格或"人设"，它需要的是一份清晰的契约（contract）。
3. **组织中的智能密度不均匀**。不同空间对智能的需求差异极大——一个代码仓库需要的智能和一次站会需要的智能完全不同，因此 Agent 必须是分层的、可组合的。

## 核心论点

### 论点一：身份三级——地基 / 顾问 / 成员

组织中的 AI 身份不是二元的（有/没有），而是三级的：

- **地基 Agent（Infrastructure）**：组织的底层能力——记忆、调度、验证。它们不服务于特定任务，而是为所有任务提供公共基础设施。类比操作系统的内核。
- **顾问 Agent（Advisor）**：被特定空间召唤的领域专家。它们有明确的技能边界，按需激活，用完不销毁而是进入待机。类比 libc 中的动态链接库。
- **成员 Agent（Member）**：持久存在于某个协作空间、承担持续职责的智能体。它们有状态、有记忆、有责任范围。类比一个长期运行的服务进程。

三级身份的核心区分维度是**生命周期**和**状态依赖**，而非"能力高低"。

### 论点二：分身机制——不销毁，待机可唤醒

传统做法：任务完成 → Agent 销毁。问题：执行过程中积累的上下文、判断依据、失败经验全部丢失。

织境的做法：任务完成 → Agent 进入待机（standby）状态。待机不是冻结，而是将其执行记忆合并到空间记忆中，自身退化为一个轻量索引。当相似任务再次出现时，通过索引匹配唤醒，而不是从零创建。

这一机制的直接收益：**组织学习的边际成本趋近于零**。第 N 次执行同类任务时，启动成本不是 O(1)，而是 O(1/N)——因为前 N-1 次的经验已经沉淀。

### 论点三：生命周期管理——创建→执行→合并→待机

```
[触发条件满足] → 创建(绑定契约) → 执行(产出结果) → 合并(经验→空间记忆) → 待机(等待唤醒)
                                                                        ↑_________|
```

关键设计约束：

- **创建阶段**必须绑定契约，不允许存在"没有职责的 Agent"。
- **合并阶段**是强制的，不允许跳过。跳过合并等同于在内存中 new 了对象但不赋值给任何引用——GC 会回收，但组织不会。
- **待机不是终点**。待机 Agent 可以被重新激活（状态恢复），也可以被显式退役（retire，经验保留，进程终止）。

### 论点四：契约驱动分层架构

每个 Agent 的行为边界由契约定义，而非由提示词（prompt）定义。契约是结构化的、可验证的、可组合的：

- **输入契约**：这个 Agent 需要什么信息才能开始工作？
- **输出契约**：这个 Agent 产出的结果满足什么格式和质量标准？
- **状态契约**：这个 Agent 在执行过程中会修改哪些共享状态？

契约的存在使得 Agent 之间可以**在编译期检查兼容性**，而非在运行时通过错误发现不匹配。

## 理论论证

**Codex 源码验证**。OpenAI Codex 的 Agent 架构本质上是一个"创建→执行→合并"的循环：每个 code session 创建一个沙盒环境，执行代码变更，然后通过 git diff 将结果合并回主分支。Codex 的核心洞察是——Agent 的价值不在于单次执行的准确率，而在于**执行轨迹的可追溯性和可复用性**。

**Claude Code 架构对比**。Anthropic 的 Claude Code 采用了类似的"工具调用链"模型，但差异在于状态管理：Codex 偏向无状态（每次调用是独立的），Claude Code 偏向有状态（会话内维持上下文）。织境的三级身份体系试图兼容两者——地基 Agent 无状态，成员 Agent 有状态。

**契约理论的软件工程基础**。Design by Contract（Bertrand Meyer, 1986）已经在对象 oriented 编程中被验证了 40 年。将其从类级别提升到 Agent 级别，逻辑上是自然的延伸。

## 与其他模块的关系

- **依赖 06-记忆系统**：Agent 的合并阶段和唤醒机制完全依赖记忆系统的分层存储和索引能力。
- **依赖 07-验证门控**：Agent 产出的知识必须通过验证门控才能被标记为"可信赖的"并写入空间记忆。
- **被 08-空间智能化依赖**：空间智能化模块定义"每个空间都有 Agent"，本模块定义"Agent 本身是什么"。

### 论点五：Agent 间通信协议——基于版本化状态的异步协调

**问题本质**：Agent 不是孤岛。主 Agent 与子 Agent、同公会内的多个 Agent、跨组织的 Agent 之间，都需要共享状态、同步知识、协商变更。这不是一个"消息传递"问题，而是一个**分布式状态一致性**问题。

#### 核心模型：版本化同步仓库

Agent 之间的共享记忆不是"你发我一段文本快照"，而是维护一个类似 Git 的**版本化同步仓库（Sync Repo）**：

```
Agent A（主）              Sync Repo              Agent B（子）
    │                    ┌──────────┐                  │
    ├── init ──────────→ │ v0: 初始状态 │ ←───────── init ──┤
    │                    └──────────┘                  │
    │                         │                        │
    │                    ┌──────────┐                  │
    ├── push(diff) ────→ │ v1: A的变更 │                  │
    │                    └──────────┘                  │
    │                         │                        │
    │                    ┌──────────┐                  │
    │                    │ v2: B的变更 │ ←─────── push(diff) ──┤
    │                    └──────────┘                  │
    │                         │                        │
    ├── pull ────────────  v2  ─────────── pull ───────┤
```

#### 四种原语

| 原语 | 语义 | 适用场景 |
|------|------|---------|
| `init` | 创建同步仓库，交换初始状态引用 | 主 Agent 创建子 Agent 时；两个组织建立合作关系时 |
| `push` | 提交变更（diff + 意图描述），不要求对端即时响应 | Agent 在执行中产生新的知识或状态变更 |
| `pull` | 拉取对端变更，执行 merge 或标记冲突 | Agent 主动同步对端的变化 |
| `reconcile` | 冲突解决（主裁定 / 协商 / 仲裁） | 两端对同一内容做了不同修改 |

#### 两种域模型

**域内通信（主子 / 同公会）**：

- 共享同步仓库，diff 粒度同步
- **异步执行 + 最终一致性**：子 Agent 先 push 自己的变更继续推进，不阻塞等待主 Agent 确认。主 Agent 后续 pull、review、merge 或 reject
- 主 Agent 拥有 merge 权（最终裁定），子 Agent 拥有 push 权（发起变更）
- 变更通知不要求即时响应，通过定期 pull 或事件通知触发同步

**跨域通信（不同公会 / 不同组织）**：

- 创建**共享域**（中立同步仓库），双方各自 push/pull
- 权限按读写分离：定义哪些内容只读、哪些可修改、修改后的通知方式
- 冲突无法由单方裁定时，走协商或映射到治理层（参见模块 12）的仲裁机制

#### 冲突合并策略

当两个 Agent 基于同一版本 v1 各自产出不同的 v2' 和 v2'' 时：

| 策略 | 适用场景 | 代价 |
|------|---------|------|
| **主 Agent 裁定** | 主子关系，主看两个 diff 决定用哪个或合并 | 主 Agent 的 merge 成本 |
| **双版本共存** | 平等 Agent 间，暂时保留两个版本，标记冲突 | 下次交互时必须解决 |
| **语义合并** | Agent 理解两个变更的意图，尝试自动合并 | 最理想但最复杂，需要语义理解能力 |
| **Last-write-wins** | 低价值数据，简单粗暴 | 可能丢失信息 |

在主子关系中，"主裁定"是默认策略。在平等 Agent 之间，需要引入仲裁机制。

#### 可靠性保障：心跳校验

通知可能丢失。Agent 之间需要**定期比对版本号**（而非依赖单次通知）：

```
每隔 N 次心跳：
  检查本地版本号 vs 同步仓库最新版本号
  如果不一致 → 触发 pull
  如果持续不一致超过阈值 → 升级为显式通知
```

这确保了即使通知通道不可靠，Agent 间的状态最终也会收敛。

#### 与其他模块的关系

- **06-记忆系统**：同步仓库本质上是记忆系统的"跨 Agent 共享层"。L1 索引可以跨 Agent 共享，L2 完整记录按权限控制
- **12-治理**：跨域 Agent 的冲突解决直接映射到治理模块的调解→仲裁机制
- **08-空间智能**：同一空间内的多个 Agent 共享该空间的同步仓库，空间本身就是同步仓库的边界
- **04-人机共生**：伴生 Agent 与主人之间的状态同步是这个协议的最小实例

## 开放问题

1. **同步仓库的粒度**。整个记忆作为仓库太大，单条记录作为仓库太小。最优粒度是什么？与 L1 索引的粒度对齐？还是按"主题"划分？
2. **待机 Agent 的衰退机制**。长期未被唤醒的 Agent 应该逐渐降低优先级还是保持不变？如果空间的需求发生了根本变化，旧 Agent 应该被退役还是被迁移？
3. **契约的版本管理**。当组织演进时，旧契约如何升级？Agent 是否需要支持"契约迁移"？
4. **跨平台同步仓库的可移植性**。如果两个 Agent 运行在不同平台（如 Coze vs 飞书 vs 本地），同步仓库如何实现？是否需要一个中间序列化格式？

## 可证伪条件

1. 如果在实际部署中，三级身份体系被证明是过度设计——两级（临时/持久）就足够覆盖所有场景——那么"地基/顾问/成员"的区分就是伪需求。
2. 如果分身机制（待机唤醒）在实际使用中的命中率低于 20%，说明组织的任务模式不具备足够的重复性，分身机制就是在浪费存储和索引资源。
3. 如果契约驱动在实践中被证明增加了 Agent 创建的摩擦力，导致用户倾向于绕过契约直接写提示词，那么契约的粒度设计就是错的。

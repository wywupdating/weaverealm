> 创建时间：2026-06-07
> 目的：调研GitHub上以"个人知识体系/理论框架"形式发布的成功仓库，为我们的AI工会理论仓库提供参考

## 一、参考仓库分析（6个典型）

### 1. sindresorhus/awesome — 精选列表鼻祖
- **链接**: https://github.com/sindresorhus/awesome
- **Star**: 300k+（GitHub星标数最多的非代码类仓库之一）
- **性质**: 资源精选集合（非原创理论）
- **结构**: 单个README.md，按领域分大类，每个条目一行描述+链接
- **发布节奏**: 持续增量更新，社区PR驱动
- **关键设计**:
  - Awesome Manifesto：明确"只收录真正awesome的"标准
  - 每条必须解释"为什么awesome"
  - 宁可遗漏不可滥收
  - awesome-lint自动化校验格式
  - 贡献指南（CONTRIBUTING.md）非常严格
- **可借鉴**: 质量标准声明、贡献机制、自动化校验
- **不适用**: 我们是原创理论，不是资源聚合

## 二、三种典型发布模式

### 模式A：集中发布型
- **做法**: 先积累完整内容，一次性发布较完整的版本
- **代表**: CS-Notes（初期就有大量内容）、system-design-primer
- **优点**: 第一印象完整，容易被推荐/上trending
- **缺点**: 前期积累周期长，需要内容储备充足
- **Star增长曲线**: 发布时有一波爆发，后续靠社区传播持续增长

### 模式B：持续迭代型
- **做法**: 先发骨架和核心内容，持续补充新章节
- **代表**: applied-ml、cool-ai-stuff
- **优点**: 快速上线，边做边补，有"活着"的感觉
- **缺点**: 初期内容可能不够吸引人，第一波传播力弱
- **Star增长曲线**: 缓慢爬坡，靠长期积累

### 模式C：事件驱动型（配合博客/文章节奏）
- **做法**: 每发一篇博客/文章，同步更新GitHub仓库对应章节
- **代表**: eugeneyan（博客+GitHub联动）
- **优点**: 双渠道互相引流，每篇博文都是GitHub的流量入口
- **缺点**: 需要同时维护两个渠道
- **Star增长曲线**: 随博客发布节奏阶梯式增长

## 三、我们的推荐方案

### 推荐：模式C（事件驱动） + 模式B（持续迭代）混合

**理由**：
1. 我们已有大量原始素材（游戏×AI理论.md 118KB），不需要从零积累
2. 但素材需要"翻译"为可公开的版本，需要分批处理
3. 自媒体方案已经规划了知乎/掘金发文节奏，可以同步联动
4. GitHub做"完整底稿"，自媒体做"提炼文章"，天然互补

### 具体节奏

| 阶段 | 时间 | 动作 | 产出 |
|------|------|------|------|
| **Phase 0: 骨架** | 第1周 | 建仓库+README+目录结构+3篇核心文章 | 公开仓库上线 |
| **Phase 1: 核心填充** | 第2-3周 | 每周2-3篇，配合自媒体发文 | 6-9篇核心理论 |
| **Phase 2: 持续迭代** | 持续 | 每次有新洞察就更新对应章节 | 仓库持续生长 |

### Phase 0 发布内容（建议首批3篇）

1. **README.md** — 全景介绍+知识地图+阅读指引（最重要，决定第一印象）
2. **AI工会理论概述** — 核心模型全景（从5万字理论中提炼3000字精华）
3. **Thread-Agent模型** — 技术深度展示（吸引开发者群体）

### 首批内容缺少什么（需要补充的环节）

| 缺失环节 | 说明 | 优先级 |
|----------|------|--------|
| **可视化图表** | 架构图/流程图/关系图，纯文字太干 | P0 |
| **实际案例** | 理论有但缺少"跑了这个case"的佐证 | P0 |
| **和其他理论的对比** | 读者需要坐标系才能理解你的位置 | P1 |
| **FAQ/常见反驳** | 预判读者质疑，主动回应 | P1 |
| **贡献指南** | 即使暂不接受PR，也要有CONTRIBUTING.md | P2 |
| **版本演进记录** | 展示思考过程（CHANGELOG） | P2 |

## 五、每次发布的标准流程

```
1. 素材提取 → 从原始理论文件中提取对应章节
2. 公开化处理 → 去除敏感信息（公司名/人名/薪资等）
3. 结构化重写 → 按公开阅读习惯重新组织，不是直接搬运
4. 可视化补充 → 至少一张架构图/关系图
5. 案例填充 → 补充实际案例或思想实验
6. 交叉引用 → 和其他章节互相链接
7. README更新 → 知识地图同步更新
8. CHANGELOG记录 → 版本迭代日志
9. 自媒体联动 → 对应博文发布，文末指向GitHub
```

## 六、避坑清单

| 坑 | 说明 | 预防 |
|---|------|------|
| 一次全发 | 内容太多没人看，且暴露未完善的逻辑 | 分批发布，首批3篇 |
| 直接搬原始笔记 | 原始文件是对话体/碎片化，不适合公开阅读 | 每篇重写为独立可读的文章 |
| 只发不管 | GitHub仓库需要持续维护才有人信 | 绑定自媒体发文节奏，自然驱动更新 |
| 追求完美 | 等理论"完全想通"再发=永远不会发 | 标注"v0.x 在持续迭代"，展示演进过程反而加分 |
| 忽视README | 很多人只看README就决定要不要继续读 | README花最多时间打磨 |
| 中英双语 | ROI太低，初期不值得 | 中文为主，README做英文简介即可 |

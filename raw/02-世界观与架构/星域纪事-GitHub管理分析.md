# 星域纪事 — GitHub管理深度分析

> 目的：基于共创机制设计，梳理Git管理层面可能遇到的问题、机会和补充方案，确保项目稳定推进。

---

## 一、核心管理挑战

### 1.1 理论与故事的版本耦合

**问题**：理论在迭代，故事也在迭代。当理论v2修改了某个核心概念，引用它的3篇故事怎么办？

| 场景 | 后果 |
|------|------|
| 理论改了"三层记忆"的定义 | 引用这个概念的5篇故事里的描述全部过时 |
| 新增了一个世界观设定 | 之前的故事可能没考虑到，产生隐性矛盾 |
| 某篇故事的设定被采纳进正典 | 其他故事的描述可能和新正典冲突 |

**解决方案**：

```
每个理论文件头部声明：
---
theory_id: T02
version: 1.2
last_updated: 2026-06-07
breaking_change: false   ← 是否有破坏性变更
supersedes: T02-v1.1     ← 替代了哪个版本
---

每个故事文件头部声明依赖：
---
story_id: A
depends_theories: [T01-v1.0, T02-v1.2]  ← 依赖的理论版本
depends_worldview: v1.0                   ← 依赖的世界观版本
status: canon                            ← canon/community/draft
---
```

- 理论做breaking change时，维护者批量检查依赖它的故事，提issue通知
- 故事不会因为理论更新自动失效——标记为"基于v1.x版本"，新版理论可以共存
- 用脚本定期扫描依赖关系，生成"过时内容"报告

### 1.2 并行PR的冲突问题

**问题**：两个人同时写了不同的短篇，但都在同一个暗线节点附近，且都新增了世界观设定。

| 情况 | 例子 |
|------|------|
| 设定冲突 | A写了"验证场需要3人评审"，B写了"验证场是AI自动评审" |
| 角色冲突 | A让张三当了星尘的副会长，B让李四当副会长 |
| 时间线冲突 | A的故事发生在"星尘成立第1年"，B的也写第1年但事件对不上 |

**解决方案**：

- **WIP机制**：贡献者开PR前先开Issue占位，声明"我要写XX节点的故事，涉及XX设定"
- **30天锁定**：Issue开出来后30天内提交PR，超时释放
- **合并顺序**：先合入的为准，后合入的需要适配
- **Discord/讨论区**：并行贡献者在开写前沟通，避免撞车

### 1.3 世界观一致性的规模问题

**问题**：50篇故事 × 34个理论 × 世界观设定，人工审核一致性会越来越难。

**解决方案**：分层防御

| 层 | 做什么 | 工具 |
|---|--------|------|
| **L0：自动化检查** | CI脚本检查frontmatter完整性、依赖是否存在、格式是否合规 | GitHub Actions |
| **L1：知识图谱** | 维护一个`worldview-graph.json`，记录所有设定之间的引用关系 | 自动生成 + 人工维护 |
| **L2：人工审核** | 审核者对照图谱检查矛盾 | 维护者 |
| **L3：社区纠错** | 任何人发现矛盾可以提Issue | 读者 |

L0和L1是关键——没有自动化，规模大了维护者会被拖垮。

## 二、Git工作流设计

### 2.1 分支策略

```
main                    ← 正典（canon），受保护，只有维护者可合并
├── community           ← 社区内容（community），受保护
├── feature/story-xxx   ← 具体短篇开发分支
├── feature/theory-xxx  ← 具体理论开发分支
├── fix/worldview-xxx   ← 世界观修正
└── hotfix/             ← 紧急修正（错别字、事实错误等）
```

- main和community分支受保护，必须通过PR合并
- 正典PR需要至少1个维护者approve
- 社区PR可以由"探索者"级别以上的贡献者review

### 2.2 Release策略

| 版本类型 | 触发条件 | 示例 |
|---------|---------|------|
| **世界观大版本** | 核心设定变更，影响多篇故事 | `世界观-v2.0` |
| **故事合集** | 每5篇正典短篇出一个合集 | `正典-第一卷` |
| **理论版本** | 理论框架更新 | `理论框架-v1.2` |
| **年度总结** | 每年出一版总结 | `星域纪事-2026年报` |

Release不只是打tag——每次release同时生成一份"世界观快照"，标记所有正典内容在那一刻的状态。

### 2.3 Issue管理

用标签体系管理不同类型的贡献和问题：

| 标签 | 用途 |
|------|------|
| `story-proposal` | 短篇故事提案 |
| `theory-proposal` | 理论补充提案 |
| `worldview-conflict` | 世界观矛盾报告 |
| `content-outdated` | 内容过时（理论更新导致） |
| `good-first-issue` | 新人友好任务（适合旅人级别） |
| `canon-review` | 正典审核讨论 |
| `darkline-integration` | 暗线晋升讨论 |

### 2.4 自动化（GitHub Actions）

| 自动化 | 做什么 | 价值 |
|--------|--------|------|
| **frontmatter检查** | 提交PR时自动检查文件头是否完整、格式正确 | 防止不规范内容进入 |
| **依赖扫描** | 检查故事依赖的理论是否存在、版本是否有效 | 防止幽灵引用 |
| **目录索引更新** | PR合并后自动更新README中的目录列表 | 维护者不用手动更新 |
| **变更日志** | 自动生成"本期新增内容"摘要 | 读者快速了解更新 |
| **死链检查** | 定期扫描故事之间的交叉引用是否有效 | 防止引用断裂 |

## 三、独有问题（区别于普通开源项目）

### 3.1 "双轨质量"难题

普通开源项目：代码要么能跑要么不能跑，质量相对客观。

星域纪事：故事质量主观，理论质量客观。一个贡献可能"故事写得很好但理论有硬伤"或"理论很扎实但故事太干"。

**对策**：分离审核——故事走文学审核，理论走逻辑审核，可以一个pass一个不pass。贡献者可以选择只写故事（用已有理论）或只写理论（不配故事）。

### 3.2 "世界观债务"

每篇故事都在消耗世界观"空白"——某个角落被写死了，后续贡献者就不能自由发挥了。这像技术债务，但是叙事层面的。

**对策**：
- 世界观设定集中标记"已确定区域"和"开放区域"
- 开放区域允许贡献者自由发挥
- 已确定区域需要严格遵循既有设定
- 定期"世界观回顾"，清理过紧的约束

### 3.3 主创者依赖

项目早期高度依赖主创者（大白）的理论输出和世界观维护。如果大白一段时间没推进，社区会停摆。

**对策**：
- 维护者不只大白一人——Phase 1就培养2-3个核心维护者
- 理论体系的文档足够清晰，维护者可以独立审核理论质量
- 社区短篇不依赖主创者推进——贡献者可以在已有世界观内自由创作

### 3.4 "抢主角"问题

贡献者可能想写暗线主角的故事，但暗线是受限的。

**对策**：
- 明确规则：暗线只有核心维护者可以写
- 但贡献者可以写"和主角擦肩而过"的社区短篇——主角作为背景板出现
- 如果社区短篇中主角的描写和暗线框架一致，可以晋升

## 四、你可能没想到的好处

### 4.1 Git历史 = 世界观编年史

每一次commit都是世界观的一个变化节点。读者可以回溯到任何一个版本，看世界观是怎么一步步演化过来的。这本身就是一种叙事——"星域纪事的世界是怎么被创造出来的"。

### 4.2 Issue讨论 = 虚构世界的"议会"

Issue不只能用来报bug——可以用来讨论"星域中某个规则是否合理"。这种讨论本身就是在丰富世界观。甚至可以故意留一些"争议设定"，让社区通过Issue辩论来决定走向。

### 4.3 Fork = 平行宇宙

有人fork了仓库但走了完全不同的方向？那就是"平行星域"——和正典共享基础设定但发展出不同的故事。这不需要被阻止，反而可以被正典引用为"多元宇宙"的一部分。

### 4.4 Star和Watch = 真实的影响力指标

不用猜读者喜欢什么——GitHub的star、watch、fork数据直接告诉你哪些故事和理论最受欢迎。这比自媒体平台的数据更真实（没有算法推荐干扰）。

### 4.5 CI/CD = 世界观一致性保障

前面说的自动化检查，本质上是给"世界观不崩塌"这个约束装了一套自动测试。这在传统小说创作中根本做不到——开源+结构化+自动化，让"多人共创一个不崩的世界"变得可行。

### 4.6 贡献图 = 可视化的创作地图

GitHub的contribution graph天然适合展示"这个世界在哪些角落被丰富了"。可以做一个可视化页面，把贡献图映射到星域地图上。

## 五、补充建议

### 5.1 增加一个 `meta/` 目录

```
meta/
├── worldview-graph.json      ← 世界观实体关系图谱（机器可读）
├── dependency-matrix.md      ← 故事×理论×设定的依赖矩阵
├── contribution-stats.md     ← 贡献统计（自动生成）
└── governance.md             ← 治理规则（本文件的一部分）
```

这个目录维护者用，不面向普通读者，但对长期管理至关重要。

### 5.2 定期"世界观对齐"

每季度做一次全面对齐：
1. 扫描所有正典内容，检查隐性矛盾
2. 更新worldview-graph.json
3. 发布"世界观状态报告"
4. 标记需要更新或废弃的内容

### 5.3 理论的可独立引用

每篇理论文档应该有独立的DOI或持久标识符（可以用GitHub的permalink）。这样外部引用时不怕版本变化。理论文档内部的章节编号要稳定——不能因为新增内容就把所有编号打乱。

### 5.4 许可证选择

| 方案 | 优点 | 缺点 |
|------|------|------|
| **CC BY-SA 4.0** | 允许自由使用和再创作，要求署名和同协议共享 | 理论部分可能被商业化使用 |
| **CC BY-NC-SA 4.0** | 禁止商业使用 | 限制了传播和衍生 |
| **双许可** | 理论用MIT（鼓励使用），故事用CC BY-SA | 管理复杂 |

建议：**理论部分用MIT**（鼓励实际应用和引用），**故事部分用CC BY-SA 4.0**（允许再创作但要求共享）。在CONTRIBUTING.md中明确声明。

### 5.5 "故事审核清单"

给审核者一份检查清单，降低审核负担：

```
□ frontmatter完整且格式正确
□ 故事中的设定在worldview.md中有依据（或mapping.md中声明了新增）
□ 角色行为和character-atlas.md中的描述一致
□ 理论部分不依赖故事也能独立理解
□ mapping.md说明了暗线关联
□ 无明显逻辑漏洞
□ 语言通顺，无明显错别字
```

## 六、风险总结

| 风险 | 严重度 | 概率 | 缓解措施 |
|------|--------|------|---------|
| 世界观矛盾积累到不可修复 | 高 | 中 | 自动化检查 + 季度对齐 + worldview-graph |
| 维护者精力不足导致审核积压 | 高 | 高 | 培养多个维护者 + 自动化L0层 |
| 贡献者流失，社区冷清 | 中 | 高 | 冷启动策略 + 低门槛good-first-issue |
| 理论更新导致故事大面积过时 | 中 | 中 | 版本化依赖 + 过时内容自动检测 |
| 抢主角/抢暗线节点 | 低 | 中 | 明确规则 + 暗线只限核心维护者 |
| 许可证纠纷 | 中 | 低 | CONTRIBUTING.md明确声明 + 双许可方案 |
| IP被盗用/未署名 | 中 | 低 | Git历史天然记录贡献 + CC协议 |

## 七、结论

GitHub管理这套共创体系，最大的难点不是技术（分支、合并、CI这些都很成熟），而是**叙事一致性管理**和**社区治理**。

技术上，核心投入应该在：
1. **依赖追踪系统**（理论↔故事↔世界观的版本关系）
2. **自动化检查**（CI pipeline，降低人工审核负担）
3. **世界观图谱**（机器可读的实体关系，支持一致性检查）

治理上，核心投入应该在：
1. **早期培养2-3个核心维护者**（不能只靠大白一个人）
2. **审核流程标准化**（检查清单 + 分层审核）
3. **贡献者成长路径清晰**（旅人→探索者→铸造师→架构师）

一句话：**把世界观当代码管，把故事当PR审，把理论当依赖追踪**。

## 八、三层架构带来的管理变化

> 以下是对原文档的补充。三层架构（核心层/中间层/外层）改变了项目的管理重心。

### 8.1 管理精力的分配

| 层级 | 投入精力 | 原因 |
|------|---------|------|
| **核心层** | 70% | 这是项目的根基，必须保证真实性和可验证性 |
| **中间层** | 25% | 审核核心知识准确性，但允许叙事润色 |
| **外层** | 5% | 基本把关即可，不需要精细审核 |

**关键转变**：从"对所有内容一视同仁地审核"变成"核心严控，外层放养"。维护者的精力应该集中在核心层。

### 8.2 外层的自驱机制

外层不需要维护者花太多精力，但要设计自驱机制：

- **Star排名**：外层内容按star数排序，好作品自然浮上来
- **读者标签**：读者可以用Issue给外层作品打标签（"设定很棒""剧情炸裂""核心知识有误"等）
- **自动索引**：CI自动生成外层作品目录，按热度排序
- **核心关联度标记**：自动检测外层作品引用了哪些核心层内容，方便读者溯源

### 8.3 核心层的特殊保护

核心层是整个项目最珍贵的部分。额外保护措施：

- **变更需要RFC流程**：不只是PR，而是RFC（Request for Comments）——先写提案讨论，达成共识后再改
- **不可删除原则**：核心层内容只能被"新版本替代"，不能直接删除。旧版本标记为deprecated但保留
- **独立版本号**：核心层每个理论有独立的语义版本号（如T01 v1.2.0），变更可追溯
- **引用溯源**：中间层和外层引用核心理论时，必须标注具体版本号

### 8.4 三层对Git工作流的影响

```
main分支策略调整：
├── main                    ← 只包含核心层已发布内容
├── knowledge-main          ← 中间层内容，和main同步更新
├── outer-main              ← 外层内容，独立更新节奏
└── 三条分支互相独立但单向引用
    ├── outer 可以引用 knowledge 的内容
    ├── knowledge 可以引用 core 的内容
    └── 反向不行：core 不依赖 knowledge 或 outer
```

这个单向依赖非常重要——核心层永远不会因为外层的叙事需求而改变。理论就是理论，不会因为"这样写故事更好看"就调整。

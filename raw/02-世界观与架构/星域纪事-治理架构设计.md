# 星域纪事 — 治理架构设计

> 核心问题：发起者不是永远的核心，项目必须有让真正的大牛进来并拥有话语权的机制。
> 目标：让对的人在对的位置上有对的权力，既不集权也不混乱。

---

## 一、角色定义与权力范围

### 1.1 五种角色

| 角色 | 谁来当 | 权力 | 任期 |
|------|--------|------|------|
| **发起人（Founder）** | 大白 | 早期全权决策，后期逐步让渡 | 终身荣誉，但不终身掌权 |
| **核心维护者（Core Maintainer）** | 由社区产生 | 核心层的审核权、合并权 | 滚动制，活跃即留任 |
| **领域专家（Domain Expert）** | AI大牛/资深实践者 | 核心层理论的否决权和建议权 | 按领域认证，不限数量 |
| **叙事顾问（Narrative Advisor）** | 专业小说作者/编剧 | 中间层和外层的叙事质量话语权 | 按作品认证 |
| **贡献者（Contributor）** | 任何人 | 提交PR、参与讨论、提出Issue | 无门槛 |

### 1.2 权力矩阵

| 决策类型 | 发起人 | 核心维护者 | 领域专家 | 叙事顾问 | 贡献者 |
|---------|--------|-----------|---------|---------|--------|
| 核心层理论合并 | ✅ | ✅（需2人） | ✅ 否决权 | ❌ | 提议 |
| 核心层理论否决 | ✅ | ✅（需2人） | ✅（1人即可） | ❌ | 提议 |
| 世界观核心设定变更 | ✅ | ✅（需2人） | 建议 | 建议 | 提议 |
| 中间层内容审核 | ❌ | ✅（1人） | 建议 | ✅（1人） | 提议 |
| 外层内容审核 | ❌ | ❌ | ❌ | ❌ | 自治（基本把关即可） |
| 新核心维护者提名 | ✅ | ✅ 提名权 | ✅ 推荐权 | ❌ | 推荐 |
| 项目方向重大调整 | ✅ 发起讨论 | ✅ 投票 | ✅ 投票 | ✅ 投票 | 讨论 |

**关键设计**：
- 发起人（大白）在核心层有合并权，但**领域专家有单独的否决权**——一个大牛说"这个理论有问题"，就可以block合并
- 中间层的审核权开放给叙事顾问——让专业的人审专业的事
- 外层几乎完全自治——不需要任何人审批，基本把关就行
- 发起人**没有**中间层和外层的审核权——不越权

## 二、发起人的角色演变

### 2.1 四个阶段

```
Phase 0：独裁期（现在 ~ 5篇短篇完成前）
├── 大白 = 全权决策者
├── 所有内容自己审核
├── 快速迭代，不需要等任何人
└── 目标：把骨架搭起来，让别人看到"这个项目有东西"

Phase 1：合伙期（5篇短篇后 ~ 20篇短篇）
├── 大白 = 首席维护者 + 引路人
├── 开始放权：接受第一批核心维护者
├── 重大决策开始征求社区意见
├── 但核心层最终决定权仍在发起人
└── 目标：建立初始治理规则，让第一批贡献者有归属感

Phase 2：共治期（社区贡献稳定后）
├── 大白 = 治理者之一（不再是唯一决策者）
├── 核心层合并需要2个核心维护者approve
├── 领域专家获得否决权
├── 发起人的特权：发起重大方向讨论、否决明显有害的内容
└── 目标：项目不再依赖任何单个人

Phase 3：生态期（项目自驱运转）
├── 大白 = 精神发起人（Founder Emeritus）
├── 可以完全不管项目，项目照样运转
├── 发起人保留：核心层RFC的优先讨论权、项目精神方向的发言权
├── 不保留：任何单独的合并权或否决权
└── 目标：项目属于社区，不属于任何人
```

### 2.2 放权的时间触发器

不是按时间放权，是按里程碑：

| 里程碑 | 放什么权 |
|--------|---------|
| 第一个非发起人的核心维护者产生 | 中间层审核权下放 |
| 3个核心维护者在位 | 核心层合并需要2人approve（含发起人不超过1票） |
| 第一个领域专家认证 | 核心层理论获得专家否决权机制 |
| 核心维护者达到5人 | 发起人退出日常审核，只参与RFC和重大方向 |
| 社区月活跃贡献者超过20人 | 发起人进入Emeritus状态 |

### 2.3 发起人的底线权力（终身保留）

即使进入Emeritus状态，发起人保留以下权利：
1. **项目精神否决权**：如果社区要把项目带向违背初衷的方向（比如变成纯文学项目、商业化失控等），发起人可以否决
2. **RFC优先讨论权**：任何核心层的RFC，发起人有权第一个发表意见
3. **项目重启权**：如果社区分裂，发起人有权fork项目以保留原始精神

这三项权力不参与日常管理，只在极端情况下使用。类似宪法法院——平时不管事，关键时刻有最终裁决权。

## 三、核心维护者的产生与退出

### 3.1 产生机制

**路径A：贡献晋升（最常见）**
```
贡献者 → 3篇中间层或1篇核心层 → 被提名 → 现有维护者投票（过半数）→ 试用期3个月 → 正式任命
```

**路径B：专家直聘（针对大牛）**
```
领域专家（已有行业影响力的人）→ 发起人或2个维护者邀请 → 直接任命为维护者 → 试用期3个月 → 正式
```

### 3.2 核心维护者的责任

| 责任 | 频率 |
|------|------|
| 审核PR（按自己负责的层级） | 持续 |
| 参与RFC讨论 | 按需 |
| 维护自己负责的模块/领域 | 持续 |
| 参与社区讨论和Issue回复 | 每周 |
| 参与维护者月度同步 | 每月 |

### 3.3 退出机制

| 情况 | 处理 |
|------|------|
| 主动退出 | 随时可以，保留荣誉头衔 |
| 连续3个月无活动 | 自动转为"休眠维护者"，权限降级 |
| 连续6个月无活动 | 自动转为荣誉贡献者，失去合并权 |
| 社区投诉 + 2/3维护者投票通过 | 免职 |
| 严重违规（造假、恶意破坏） | 发起人直接免职 |

### 3.4 核心维护者的领域分工

维护者不是什么都管，而是按领域分工：

| 领域 | 负责什么 | 谁适合当 |
|------|---------|---------|
| **理论核心** | 核心层理论的审核和合并 | AI架构师、AI工程化实践者 |
| **世界观一致性** | 跨层世界观矛盾检测 | 逻辑缜密、熟悉全部设定的深度读者 |
| **中间层叙事** | 知识短篇的质量和准确性 | 有技术背景+文字能力的人 |
| **社区运营** | Issue管理、新人引导、讨论区维护 | 有社区运营经验的人 |
| **基础设施** | CI/CD、自动化脚本、仓库维护 | DevOps/工具链开发者 |

**一个维护者可以负责多个领域，一个领域可以有多人负责**。

## 四、领域专家机制

### 4.1 为什么要单独设这个角色

核心层理论的真实性是项目的命根。核心维护者可能很认真但不是某个领域的专家。需要一个机制让真正的专家能对理论质量有一票否决。

### 4.2 领域专家的认证

| 条件 | 满足任一即可 |
|------|------------|
| 在相关领域发表过论文/有被引用的技术文章 | 学术认证 |
| 在相关领域有5年以上工程实践 | 实践认证 |
| 相关领域知名开源项目的核心贡献者 | 开源认证 |
| 被至少2个现有领域专家推荐 | 同行认证 |

### 4.3 领域专家的权力

- **理论否决权**：对核心层PR，领域专家可以单独行使否决（必须附书面理由）
- **理论建议权**：对任何层的内容提出专业意见，维护者应优先处理
- **RFC发起权**：可以发起核心层的RFC
- **不包含合并权**：专家只管"对不对"，不管"要不要"——合并仍是维护者的事

### 4.4 专家否决的流程

```
领域专家行使否决
  → 在PR中标记 "expert-veto" + 书面理由
  → 维护者不能合并该PR
  → 贡献者根据理由修改
  → 修改后同一专家或另一专家确认
  → veto解除，恢复正常审核流程
  → 如果贡献者不同意否决 → 发起RFC讨论 → 维护者集体裁决
```

## 五、叙事顾问机制

### 5.1 叙事顾问的认证

| 条件 | 满足任一即可 |
|------|------------|
| 出版过小说或有连载完结作品 | 出版认证 |
| 获得过征文/写作类奖项 | 奖项认证 |
| 在平台有稳定的读者基础（订阅/追更/评分） | 读者认证 |
| 被至少2个现有叙事顾问推荐 | 同行认证 |

### 5.2 叙事顾问的权力

- 中间层短篇的审核权（1人即可approve）
- 外层作品的推荐权（推荐的作品在README中展示）
- 跨篇叙事体系的设计权
- 可以发起"叙事提升提案"——对已发布的作品提出改版建议

### 5.3 叙事顾问不碰核心层

叙事顾问的权限严格限制在中间层和外层。核心层理论的真实性不是叙事顾问的职责范围。如果叙事顾问同时也是AI专家，可以同时获得领域专家身份。

## 六、决策机制

### 6.1 日常决策（不需要投票）

- PR审核：按层级对应的审核流程
- Issue处理：任何维护者可以响应
- 世界观微调：不涉及核心理论的，维护者可直接决定

### 6.2 重大决策（需要投票）

| 决策类型 | 触发条件 | 投票范围 | 通过条件 |
|---------|---------|---------|---------|
| 核心层RFC | 核心理论变更 | 核心维护者 + 领域专家 | 2/3通过 |
| 新核心维护者 | 提名后 | 现有核心维护者 | 过半数 |
| 项目方向调整 | 发起人或3个贡献者提议 | 全社区 | 2/3通过 |
| 治理规则变更 | 任何维护者提议 | 核心维护者 | 2/3通过 |
| 社区投诉裁决 | 投诉提交后 | 核心维护者（回避当事人） | 2/3通过 |

### 6.3 投票周期

- 投票期7天
- 投票期间任何人可以发表意见
- 结果在Issue中公示

## 七、知识产权与收益

### 7.1 署名规则

| 层级 | 署名方式 |
|------|---------|
| 核心层 | 理论原作者（Git历史记录） |
| 中间层 | 理论贡献者 + 叙事优化者（如适用） |
| 外层 | 完全归原作者 |

### 7.2 如果未来有商业收益

| 场景 | 分配原则 |
|------|---------|
| 核心理论被商业引用 | MIT协议，无需分润，但要求署名 |
| 小说出版/IP衍生 | 归原作者，但需声明"基于星域纪事世界观" |
| 世界观本身被商业使用 | 社区讨论决定，发起人有最终裁决权 |

### 7.3 CLA（贡献者许可协议）

所有贡献者在首次提交PR时需要签署CLA：
- 确认贡献内容为原创或已获授权
- 确认接受项目的许可证（理论MIT，故事CC BY-SA）
- 确认理解三层架构的治理规则

## 八、反腐败机制

### 8.1 防止发起人独裁

- Phase 2后，发起人没有单独的核心层合并权
- 发起人的"精神否决权"只在极端情况使用，每次使用需要在社区公示理由
- 如果社区2/3投票认为发起人滥用权力，可以启动"发起人弹劾"流程

### 8.2 防止核心维护者小圈子

- 核心维护者的产生需要公开投票
- 任何贡献者可以提名新维护者
- 连续3个月无活动的维护者自动降级
- 维护者的所有决策在Issue/PR中公开，不可私下交易

### 8.3 防止领域专家滥用否决权

- 否决必须附书面理由
- 同一专家对同一PR只能否决一次
- 贡献者可以申请RFC讨论来挑战否决
- 如果专家频繁否决且理由不充分，其他专家/维护者可以提出质疑

### 8.4 防止外层内容失控

- 外层虽然审核宽松，但有底线：不违法、不恶意、不曲解核心理论
- 三次违反底线的外层作者会被标记，后续提交需要维护者审核
- 严重违规（抄袭、恶意内容）直接ban

## 九、治理文档本身的管理

本文件（治理架构设计）属于项目的"宪法"级文档：

- 放在仓库根目录，命名为 `GOVERNANCE.md`
- 修改需要2/3核心维护者投票通过
- 发起人有一票否决权（仅限治理文档的修改）
- 每半年回顾一次，看是否需要调整

## 十、总结：权力流动的方向

```
发起人独占权力
  → 贡献者增多 → 核心维护者产生 → 权力分散
  → 领域专家加入 → 核心层质量有专业保障
  → 叙事顾问加入 → 中间层和外层质量提升
  → 发起人逐步退到"精神领袖"位置
  → 项目属于社区

理想状态：
- 没有任何人可以单独决定核心理论的方向
- 但每个人都有机会影响核心理论的方向
- 发起人的价值不是"永远掌权"，而是"搭建了一个让对的人能做对的事的系统"
```

**大白，你的角色不是永远当CEO，而是写好这个项目的宪法，然后让最好的法官来执法。**

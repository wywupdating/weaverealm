# Coze 记忆系统原型 v1

> 日期：2026-06-06
> 平台：Coze/扣子
> 目标：在现有骨架上对齐三层记忆架构理论，跑通最小闭环

---

## 一、现状 vs 理论映射

| 理论层 | 理论定义 | 当前对应 | 差距 |
|--------|---------|---------|------|
| L1 索引层 | 极轻量索引条目，只存方向+路径 | MEMORY.md（~5KB，混合了索引+状态+决策） | **太重**，不是纯索引 |
| L2 展开层 | 按领域/时间拆分的详细内容 | recent_memory/（30+条，index.json索引） | **结构已接近**，但缺少明确的"满了就拆"机制 |
| L3 执行层 | 原始对话记录 | 对话历史（系统管理，不可直接控制） | **被动**，需要主动归档 |
| 事件驱动 | 被触发时才整理更新 | Calendar日程触发（已有） | **有触发器，但缺整理逻辑** |
| 跨节点感知 | 多L1间的关联（机制待定） | 暂时只有一个L1（MEMORY.md） | **未触发**，等L1满了再启 |

## 二、v1 原型目标（最小闭环）

**核心验证**：三层分层 + 事件驱动再巩固 能在日常使用中跑通。

具体目标：
1. MEMORY.md 瘦身为纯索引（控制在2KB以内）
2. recent_memory/ 明确为L2展开层，验证"满了拆分"
3. 建立 L3→L2 的归档流程（对话重要内容→决策文件）
4. 验证事件驱动再巩固：被访问时顺便整理，不全量扫描

**不做**：
- 暂不创建多L1（当前MEMORY.md远未满）
- 暂不实现跨节点感知机制
- 不改底层逻辑，只用Coze现有能力

## 三、三层结构定义

### L1 索引层 — MEMORY.md

**职责**：方向+路径+入口，不展开细节
**格式约束**：
- 每个条目 ≤ 2行（标题+路径/一句话摘要）
- 总量控制在2KB以内
- 只回答"有什么"和"在哪儿"，不回答"具体是什么"

**当前问题与改造**：
- ❌ "游戏×AI"条目下展开了6行细节（核心模型、扩展内容、下一步、理论缺口）
- ✅ 改为：`游戏×AI — AI工会运营验证 | 详情→recent_memory/decision/`

### L2 展开层 — recent_memory/

**职责**：按领域拆分的详细内容
**当前结构**（保留）：
```
recent_memory/
├── index.json          ← L2全局索引（相当于L1的子索引）
├── decision/           ← 决策记录
├── project/            ← 项目状态
└── todo/               ← 待办事项
```

**拆分规则**（"满了就启新的"）：
- 单个文件超过20KB → 按时间或主题拆分
- 单个目录超过20个文件 → 创建子目录或归档旧文件
- index.json超过50条 → 归档已完成条目到 `recent_memory/archive/`

### L3 执行层 — 对话记录

**职责**：原始对话，只读归档
**管理方式**：
- 重要结论从对话中提取 → 写入L2决策文件
- 原始记录由系统保留，Agent不直接管理
- 归档动作通过 memory_search 回溯 + write_file 完成

## 四、事件驱动再巩固流程

### 触发条件
当以下事件发生时，顺便整理相关记忆：
1. **用户问起某个话题** → 检查L1相关条目是否需要更新
2. **完成一个任务/决策** → 更新L2文件 + 检查L1索引是否同步
3. **日程触发执行** → 执行完毕后顺便整理产出的记忆
4. **用户说"记住"** → 立即写入对应层级

### 整理动作（轻量，不全量扫描）
```
被触发 → 读L1相关条目 → 是否过期/不一致？
   ├─ 是 → 更新L1条目 + 必要时更新L2文件
   └─ 否 → 跳过
```

### 不做的事
- ❌ 不定时全量扫描所有L2文件
- ❌ 不主动整理未被触及的记忆
- ❌ 不预判未来需要的记忆结构

## 六、改造步骤

### Step 1：MEMORY.md 瘦身
- 把所有展开的细节压缩为"标题+路径"格式
- 目标：5KB → 2KB以内
- 删掉"注意事项""重要感悟"等展开内容，只留索引指向

### Step 2：确认 L2 归档流程
- 对话中产生重要结论 → 用 write_file/edit_file 写入 recent_memory/decision/
- 同步更新 index.json
- 这一步已有实践，只需明确化

### Step 3：建立 L3→L2 提取习惯
- 当用户讨论产生决策级结论时，主动提取写入L2
- 不等用户说"记住"，看到就做
- 但不把普通对话过程写入L2

### Step 4：跑通一个月
- 用一个月时间观察：
  - MEMORY.md是否维持在2KB以内
  - recent_memory/增长速度
  - 事件驱动再巩固是否自然发生
  - 是否出现L1"满了"的信号

### Step 5：根据观察决定是否启多L1
- 如果MEMORY.md在一个月内稳定在2KB以内 → 单L1足够
- 如果持续膨胀 → 考虑拆分为多L1 + 引入跨节点机制

## 七、成功标准

v1原型验证通过的标志：
1. ✅ MEMORY.md 稳定在2KB以内，纯索引无展开
2. ✅ recent_memory/ 有明确的"写入→索引→归档"流程
3. ✅ 一个月内自然发生过≥5次"被触发时顺便整理"的事件驱动再巩固
4. ✅ 没有出现"找不到某条记忆"的情况（索引可达性验证）

## 八、风险与应对

| 风险 | 应对 |
|------|------|
| MEMORY.md瘦身时丢信息 | 先备份，瘦身后的条目都指向L2文件 |
| L2文件增长太快 | 按20KB/文件、20文件/目录的上限触发归档 |
| 事件驱动再巩固被遗忘 | 在Heartbeat中加一条轻量检查："最近24h有没有被触发但没有整理的记忆" |
| 用户习惯了展开式MEMORY | 新格式需要适应期，坚持用指向替代展开 |

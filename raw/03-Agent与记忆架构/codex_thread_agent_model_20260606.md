# Codex架构验证 + Thread-Agent模型（2026-06-06）

> 延续三层记忆架构讨论，用Codex CLI实际实现验证理论，并提炼出Thread-Agent分层模型。

---

## 一、Codex CLI记忆架构分析

### Codex定位纠正
Codex不只是编码Agent，已扩展为**工作台**（workstation），能处理多种业务，有平台化扩张意图。

### 记忆文件体系与三层架构映射

| Codex文件 | 作用 | 对应用户三层设计 |
|-----------|------|-----------------|
| `memory_summary.md` | 启动时加载的精简摘要，5K token预算 | **L1 索引层** |
| `MEMORY.md` | 合并后的完整记忆 | **L2 展开层** |
| `raw_memories.md` | 未整理的候选记忆 | **L3→L2 中间态** |
| `rollout_summaries/` | 每个session的独立摘要 | **L3 执行层** |

### 两条压缩路径
1. **本地路径**：生成"交接摘要"（handoff note），含Goal/进度/约束/下一步，保留最近20K token用户消息原文
2. **服务端路径**：OpenAI API层加密compaction item，返回"canonical next context window"。不是文本摘要，而是模型内部状态压缩。**对模型执行效果近乎无损**（用户称之为"伪无损压缩"）

### 核心验证
- Codex写入路径=事件驱动再巩固（session空闲6小时后后台提取+合并）
- "压缩是降维不是延后"——Codex两种降维都做了（本地摘要+服务端opaque压缩）
- 用户无需切换Agent/对话窗口，单窗口无限压缩续跑

### 一个关键差异
Codex源码建议"同一线程压缩太多次后准确性会衰减，长线程不如开新线程"。**这验证了多Agent分层的必要性**——单Agent扛不住复杂线程，需要多Agent管理。

## 二、Thread-Agent模型

### 核心概念
- **Thread（线程）≈ 数据层**：持久化的记忆载体（文件/存储），不随Agent生灭
- **Agent ≈ 计算层**：可替换的处理器，负责压缩/归档/检索/再巩固

### 关键洞察
用户只看到线程被压缩/整理，看不到背后有几个Agent在干活。这就是封装。

**一句话：线程是"地"，Agent是"人"。地不变，人可以换、可以加。**

### 三层架构映射
- L1/L2/L3的文件=Thread（数据在那里，不随Agent生灭）
- Agent是挂在数据上的处理单元——压缩/归档/检索/再巩固都是Agent操作数据

### 多Agent共享线程
- 对应三层架构：L1 Agent管索引、L2 Agent管展开归档、L3 Agent管原始记录
- 共享同一套数据，各管各的层级
- 需要锁机制或职责划分（如Agent A管归档，Agent B管实时写入）

### 需要补充的边界
1. **Agent自身也有记忆**（system prompt/行为模式/处理经验），需轻量"人格配置"可继承
2. **多Agent写协调**：谁有写权限？需锁机制或职责划分
3. **迭代驱动力**：不是无目的自我循环，而是"新输入→重新理解旧记忆→更新索引/摘要"

### 与Codex对比
| 维度 | Codex | 用户设计 |
|------|-------|---------|
| Agent/线程关系 | 1个Agent管1个线程 | 多Agent管1个线程 |
| 线程满了 | 开新线程 | 加新Agent管同一线程 |
| 线程（数据） | 随线程切 | 持续，不断 |
| 可扩展性 | 受单Agent能力限制 | Agent可扩展 |

**结论：用户设计比Codex更合理——线程（数据）不断，Agent（计算）可扩展。**

## 三、与已有理论的关系

- Thread-Agent模型是三层记忆架构+分身体系的**统一抽象**
- 三层记忆=Thread的内部结构（数据视角）
- 分身体系=Agent的组织方式（计算视角）
- Thread-Agent=两者的桥梁（数据-计算分离视角）

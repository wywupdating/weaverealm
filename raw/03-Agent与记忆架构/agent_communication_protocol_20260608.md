# Agent间通信协议理论构建（06-08）

## 背景
在配置健身项目空间时，用户发现Max（主Agent）与项目Agent不共享记忆，传递上下文靠sessions_send文本注入（快照），无法持续同步。由此引发Agent间通信机制的系统性讨论。

## 核心问题
- Agent间记忆隔离（各自独立的MEMORY/USER/对话历史）
- 传递上下文是文本快照，非引用，无法跟踪变更
- 子Agent初始化后，主Agent的知识更新无法自动同步

## 用户提出的方案

### 域内（主子Agent/同公会）
- **初始化时传递文件引用**（非内容快照）
- **子Agent要改**：向主Agent提变更请求 → 主确认 → 改文件 → 双方更新
- **主Agent要改**：改文件 → 通知子Agent → 子读取即生效
- 本质：**共享文件作为唯一真相源（Single Source of Truth）**

### 跨域（不同公会/组织）
- 传递引用+对应文件内容（跨平台文件系统不可达时）
- 或引入**共享域**（中立同步区域），定义读/写权限和变更沟通方式

### Git-like diff同步（用户对反驳的回应）
- 关联时创建类似git的同步仓库，放入需要同步的文件
- 后续只看diff，不传全量
- 解决跨平台可寻址问题

### 异步协商（用户对同步阻塞的回应）
- 子改了→子按自己方案继续推进→通知主
- 主可能还在按原来的推进
- 过一阵主发现→看diff→确认改动是否合理→调整自身或通知子
- 本质：**最终一致性**，类似git的fork/PR模型

## Max补充的要点

### 冲突解决策略
1. **last-write-wins**：简单粗暴，后改覆盖先改，可能丢信息
2. **主Agent裁定**：主看两个diff决定用哪个或合并（主子关系中最合理）
3. **双版本共存**：保留两版本标记冲突，下次交互时协商
4. **语义合并**：Agent理解变更意图自动合并（最理想也最难）
- 平等Agent间需仲裁机制，可对接织境治理模块（12-governance）

### 变更通知可靠性
- 通知丢失时主用过时数据决策后果严重
- 需要心跳校验机制：定期比对版本号，发现不一致触发同步
- 与现有心跳检查思路一致，从单Agent扩展到Agent间

### 隐性知识问题
- 对话中形成的隐性知识（语气偏好、情绪感知）无法通过文件引用传递
- 用户反驳：记忆本质也是文件（不同存储形式），关注机制而非介质
- 未完全解决，留待后续探索

## 协议模型总结（v0.1）

```
核心原语：
├── init（建立同步仓库，交换初始状态）
├── push（Agent提交变更，附带diff+意图描述）
├── pull（拉取对端变更，merge或标记冲突）
├── notify（变更通知，不要求即时响应）
└── reconcile（冲突解决，主裁定/协商/仲裁）

域内（主子/同公会）：共享同步仓库，diff粒度，主有merge权，异步+最终一致
跨域（不同公会）：共享域（中立仓库），双方push/pull，冲突走协商或仲裁
```

## Coze平台当前限制
- 项目Agent和主Agent无原生共享文件机制
- 传内容靠sessions_send塞文本（非引用）
- 各自独立Calendar，日程归属创建者
- agent间无直接调度通道

### Coze现状 vs 织境模型对比（06-08分析）
| 织境模型 | Coze现状 | 差距 |
|---------|---------|------|
| init（引用传递） | sessions_send（全量文本） | ❌ 无引用机制 |
| push（diff同步） | coze agent file（全量覆盖） | ❌ 无版本/diff |
| pull（增量拉取） | 主动read（全量读取） | ⚠️ 可用但低效 |
| reconcile（冲突解决） | 后写覆盖 | ❌ 无冲突检测 |
| notify（变更通知） | 无 | ❌ 不存在 |
| 心跳校验 | heartbeat.md（单Agent内） | ⚠️ 可扩展但未原生支持 |

- **本质**：文件系统是最强共享层（无版本控制），sessions_send是通讯层（文本快照非引用），记忆完全隔离
- **同claw_id在项目空间=同一Agent的两个窗口**，不是独立Agent；记忆/对话历史完全隔离

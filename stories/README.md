# 织境故事

> 近未来，你和你的 Agent 会怎么协作？

## 定位

这些故事不是科幻。它们是把框架里的理论放进"如果明天就这样了"的场景里，让你**感受到**而不是**理解到**。

一个人读完 `framework/02-economy.md` 知道小团队的钱该怎么分，但读完一个会计在结算日对着Excel手忙脚乱的故事，他就**感受到**了。

## 写作规范

### 元数据格式

每篇故事开头必须包含以下元数据：

```markdown
---
title: 故事标题
architecture:
  - 02-economy          ← 对应 framework/ 下的模块编号
  - 11-lifecycle
status: draft | published
author: 你的名字
---

故事正文从这里开始...
```

### 写作原则

1. **近未来，不远未来** — 场景设定在1-2年内可能发生的现实，不搞星际旅行
2. **人比概念重要** — 读者记住的是角色和情境，不是术语
3. **短** — 一个场景讲透一件事，优于十页设定铺陈
4. **可以矛盾** — 不同作者对同一模块的理解可以不同，这是特性不是bug
5. **Agent有视角** — 故事可以从人的视角写，也可以从Agent的视角写，但Agent不是NPC
6. **需求先行** — 先在网上调研真实的小组织AI需求，再选框架模块匹配，确保故事有现实根基
7. **避免个人化** — 不用某个特定圈子（如摇摆舞）作为重复背景，保持场景多样性

### 不需要

- 不需要连续的剧情线
- 不需要统一的时间线
- 不需要角色一致性
- 不需要审批（PR 即可）

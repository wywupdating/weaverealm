# 织境短篇

> 架构的可选可视化窗口。

## 定位

短篇不是架构的附属品，也不是架构的翻译。短篇是独立的虚构叙事，用来**让理论可以被感知**。

一个人读完 `architecture/02-economy.md` 知道经济系统怎么运转，但读完一篇关于公会会计在结算日手忙脚乱的故事，他就**感受到**了。

## 写作规范

### 元数据格式

每篇故事开头必须包含以下元数据：

```markdown
---
title: 故事标题
architecture:
  - 02-economy          ← 对应 architecture/ 下的模块编号
  - 11-lifecycle
status: draft | published
author: 你的名字
---

故事正文从这里开始...
```

`architecture` 字段标注本篇故事涉及哪些架构模块。一篇故事可以对应一个模块，也可以对应多个。不强求覆盖所有模块，不强求一一对应。

### 写作原则

1. **理论是锚不是枷锁** — 故事要忠于架构设定的核心逻辑，但不要求面面俱到
2. **人比概念重要** — 读者记住的是角色和情境，不是术语
3. **短** — 一个场景讲透一件事，优于十页设定铺陈
4. **可以矛盾** — 不同作者对同一架构模块的理解可以不同，这是特性不是bug

### 不需要

- 不需要连续的剧情线
- 不需要统一的世界观时间线
- 不需要角色一致性（不同故事里的同名角色可以是不同的人）
- 不需要审批（PR 即可，有争议就讨论）
